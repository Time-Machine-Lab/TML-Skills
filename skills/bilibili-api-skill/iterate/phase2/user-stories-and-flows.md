# bilibili-api-skill 二期用户故事与流程设计

## 产品定位

二期的 `bilibili-api-skill` 不再只是一个零散接口集合，也不只是一个简单的命令工具。

它更像一个面向 B 站推广场景的智能操作系统。

这个产品统一提供 4 层能力：

1. 自然语言引导
2. 原子化指令调用
3. 功能包执行
4. 推广策略驱动的任务执行

也就是说，用户可以从最简单的自然语言开始，也可以逐步进入更高级的使用方式。

## 产品在用户眼里会变成什么

从用户视角看，这个产品应该呈现成下面这种样子：

- 新手可以直接问
  - “这个 skill 是干什么的？”
  - “我该怎么开始？”
- 熟手可以直接调命令
  - 比如发评论、发私信、查产品、查任务
- 进阶用户可以执行功能包
  - 比如搜视频、找意向用户、持续跟进
- 高级用户可以选择推广策略，生成完整任务流程
  - 让 agent 按一整套推广打法持续执行

所以，这个产品不是单一入口，而是一个分层使用的系统。

## 用户故事

## A 用户：新手引导型用户

### 用户状态

- 刚安装 skill
- 不清楚 skill 是做什么的
- 不清楚怎么开始
- 不知道有哪些能力

### 用户目标

- 先搞清楚这个 skill 能做什么
- 在 agent 的引导下完成第一次可用配置

### 用户行为

A 用户通常会直接用自然语言询问，比如：

- “这个 skill 是干什么的？”
- “我要怎么开始用？”
- “我需要先做什么？”

### 系统应该怎么表现

agent 不应该直接丢一堆命令，而应该分步骤引导：

1. 先解释这个 skill 的作用
2. 告诉用户第一步需要准备什么
3. 引导用户提供一个 B 站账号进行扫码登录
4. 引导用户建立第一个产品
5. 引导用户了解接下来可以怎么使用

### 这个故事的核心价值

A 用户对应的是“自然语言引导入口”。

这一层决定了新用户能不能真正进来，而不是装完就懵住。

---

## B 用户：基础命令型用户

### 用户状态

- 已经大概了解这个 skill
- 已经知道有一些基础命令可以直接调用

### 用户目标

- 快速完成一个明确动作
- 不需要进入复杂任务流程

### 用户行为

B 用户会直接做一些明确操作，比如：

- 提供某个视频 `BVID`，要求在视频下方发送评论
- 提供某个用户 ID，要求给该用户发送私信
- 查询当前有哪些产品
- 查询某个产品下有哪些任务

### 系统应该怎么表现

这一层应该以“基础命令调用”为主。

用户给出明确目标后：

- agent 识别用户意图
- 路由到对应指令
- 执行单个明确动作
- 返回清晰结果

### 这个故事的核心价值

B 用户对应的是“原子化使用入口”。

这一层决定了 skill 能不能成为一个可直接使用的工具，而不是只能跑复杂流程。

---

## C 用户：功能包型用户

### 用户状态

- 已经知道这个 skill 不只是命令调用
- 已经理解有一层“功能包”

### 用户目标

- 不想自己拼细节命令
- 希望直接完成一个阶段性的小目标

### 用户行为

C 用户会单独执行某个功能包，比如：

- 根据产品信息提炼关键词，并搜索 N 个相关视频
- 根据产品和指定视频，在评论区筛出有产品意向的用户
- 在指定时间范围内持续检查私信和评论回复，并对有意向用户做跟进

### 系统应该怎么表现

这一层不再只是执行单条指令，而是：

- 主 `Agent` 读取功能包说明
- 主 `Agent` 判断是否要把这个功能包派给副 `Agent`
- 副 `Agent` 在功能包内部组织相关指令
- 副 `Agent` 完成一个较复杂的阶段性功能后把结果交回
- 最终返回清晰结果

### 这个故事的核心价值

C 用户对应的是“阶段性功能入口”。

这一层决定了 skill 能不能从工具升级成可复用的智能功能模块。

---

## D 用户：策略型用户

### 用户状态

- 已经理解这个 skill 不只是功能包
- 已经知道 skill 里还有“推广策略”

### 用户目标

- 不只想完成一个阶段性功能
- 想让 agent 按一整套推广打法去推进任务

### 用户行为

D 用户会选择某个推广策略，比如：

- 评论引流策略
- 精准用户跟进策略
- 广撒网 + 回复承接策略

然后基于：

- 某个产品
- 某套策略
- 某些执行账号

生成一个具体任务。

### 系统应该怎么表现

这一层应该表现为：

1. 先选择产品
2. 再选择策略
3. 再生成对应的 `TaskSpec`
4. 再由主 `Agent` 按任务文档持续推进
5. 其中某些阶段可以派给副 `Agent` 独立完成

### 这个故事的核心价值

D 用户对应的是“完整推广入口”。

这一层决定了 skill 能不能从单点工具，升级成真正的推广系统。

---

## E 用户：运营与协作型用户

### 用户状态

- 不一定亲自下命令
- 更关心当前任务跑得怎么样
- 更关心哪些动作已经做过

### 用户目标

- 看进度
- 看记录
- 看结果
- 做调整

### 用户行为

E 用户会关注：

- 当前有哪些产品
- 每个产品下有哪些任务
- 某个任务当前在哪个阶段
- 已经发过哪些评论
- 已经跟进过哪些用户
- 任务有没有异常

### 系统应该怎么表现

这一层应该重点依赖：

- `TASK.md`
- `WORKLOG.md`
- `OperationRecord`

### 这个故事的核心价值

E 用户对应的是“可观察、可协作入口”。

这一层决定了这个 skill 不是黑盒，而是一个可管理的系统。

---

## 用户操作流程

从整体上看，用户操作流程可以收成 4 级。

### 第一级：自然语言进入

用户先通过自然语言了解：

- 这是什么
- 怎么开始
- 现在适合走哪种方式

### 第二级：基础命令操作

用户在明确某个动作后，可以直接做原子操作：

- 查
- 发
- 看
- 创建

### 第三级：执行功能包

用户在不想自己拼命令时，可以直接执行某个功能包，完成一个阶段性目标。

### 第四级：执行策略任务

用户在需要完整推广流程时，可以基于产品和策略生成任务，再由 agent 持续推进。

---

## 业务流程

从业务层看，这个产品的主流程是：

1. 用户接入 skill
2. 完成账号初始化
3. 建立产品
4. 选择使用方式

使用方式分成三条路：

- 直接调用基础命令
- 单独执行功能包
- 基于策略创建任务

如果用户走任务模式，后续流程就是：

1. 选择产品
2. 选择策略
3. 生成 `TaskSpec`
4. agent 持续执行
5. 执行结果沉淀到：
   - `TASK.md`
   - `WORKLOG.md`
   - `OperationRecord`

---

## 产品最终会呈现成什么样

综合上面的用户故事，这个产品最终呈现出来的样子应该是：

- 对新手友好
  - 可以直接用自然语言开始
- 对熟手友好
  - 可以直接调原子命令
- 对进阶用户友好
  - 可以直接执行功能包
- 对运营和协作友好
  - 可以基于策略和任务持续推进
  - 也可以看记录、看进度、看结果

所以，这个产品最终不是一个“单命令工具”，而是一个：

- 面向 B 站推广场景的分层智能系统

## 当前定案

- 产品面向多层用户，不只面向一种使用方式
- 用户入口分成：
  - 自然语言入口
  - 基础命令入口
  - 功能包入口
  - 策略任务入口
- `Strategy` 是推广策略模板
- `TaskSpec` 是基于产品和策略生成的具体任务文档
- `OperationRecord` 负责承接执行中的关键操作事实
