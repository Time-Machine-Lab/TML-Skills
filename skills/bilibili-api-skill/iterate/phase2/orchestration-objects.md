# bilibili-api-skill 二期编排对象设计

## 我们现在的理解

二期上层编排，先只看 3 个对象：

1. `Capability`
2. `Strategy`
3. `TaskSpec`

一句话理解：

- `Capability` 是能力包
- `Strategy` 是推广策略模板
- `TaskSpec` 是具体任务文档

agent 在运行时，按照 `TaskSpec` 去调用 `Capability`，并参考 `Strategy` 里的规则继续往下执行。

---

## 1. `Capability`

`Capability` 是封装好的高层能力。

它的重点不是底层接口，而是“agent 可以直接拿来用的一块能力”。

可以简单理解成：一个能力包就是一个稳定可调用的模块。

### 例子

- 能力 A
  - 根据产品信息提炼关键词，搜索视频，返回视频列表
- 能力 B
  - 根据产品和视频，在评论区筛出有意向的用户，返回用户列表
- 能力 C
  - 按时间间隔持续检查私信和评论回复，发现线索后触发跟进

---

## 2. `Strategy`

`Strategy` 是一套标准推广方案。

它更准确的定位是：

- 推广策略模板
- 定制化心法模板
- 一整套带判断原则的推广流程模板

它不仅回答“先做什么、后做什么”，还回答：

- 先做什么
- 后做什么
- 每一步调用什么能力
- 每一步要注意什么

所以，`Strategy` 不是一句抽象口号，也不是简单的功能包集合，而是一套带细节说明的推广打法。

### 一个 `Strategy` 至少要说明清楚什么

- 这套策略是干什么的
- 适合什么类型的产品
- 分几个阶段
- 每个阶段调用哪些 `Capability`
- 每个阶段的目标是什么
- 每个阶段什么时候进入
- 每个阶段什么时候结束
- 有哪些风控注意事项
- 有哪些动作不要做

### 你可以怎么理解它

如果用更好懂的话来说：

- `Capability` 是工具
- `Strategy` 是打法

也就是说，真正决定“任务怎么跑”的核心，其实是 `Strategy`。

### 示例

下面用一个完整例子来说明 `Strategy` 和 `TaskSpec` 是怎么配合的。

#### 例子背景

假设我们现在要推广一个产品：

- 产品：AI 编程训练营
- 目标：找到对 AI 编程、工作流、自动化感兴趣的用户
- 执行账号：账号 A、账号 B

这个时候，我们可以先选一套 `Strategy`：

- 策略名称：`评论引流 + 回复跟进`
- 策略目标：先通过公开视频评论做第一轮触达，再从互动里筛出高意向用户继续跟进

#### 这套 `Strategy` 里可以这样定义阶段

##### 阶段 0：持续监听

调用能力：

- 能力 C

阶段目标：

- 持续检查私信和评论回复
- 避免漏掉已经出现的线索

进入条件：

- 任务启动后立即开始

退出条件：

- 不退出，作为整个任务期间持续运行的后台阶段

##### 阶段 1：找视频

调用能力：

- 能力 A

阶段目标：

- 根据产品信息提炼关键词
- 搜到一批相关视频
- 形成后续可触达的视频列表

输入：

- `Product`
- 目标视频数量，比如 50 条

阶段产出：

- 一组 `BilibiliVideo`

##### 阶段 2：找意向用户

调用能力：

- 能力 B

阶段目标：

- 从阶段 1 找到的视频里，继续筛出有意向的用户

输入：

- `Product`
- 上一阶段得到的视频列表
- 目标用户数量，比如 30 人

阶段产出：

- 一组 `BilibiliUser`

##### 阶段 3：做公开视频评论触达

调用能力：

- 评论发送相关能力

阶段目标：

- 对阶段 1 得到的视频列表做第一轮公开触达

执行原则：

- 发评论前先查 `OperationRecord`
- 避免同账号对同视频重复发送相似内容
- 执行过程中参考风控限制

阶段产出：

- 一批新的 `OperationRecord`
- 成功发送后，通常会形成新的 `BilibiliComment`

##### 阶段 4：做回复或私信跟进

调用能力：

- 评论回复相关能力
- 私信发送相关能力

阶段目标：

- 对阶段 2 得到的意向用户做进一步跟进

执行原则：

- 先看用户是否已经回复过
- 再看历史上是否已经发过私信或评论回复
- 根据风控和上下文决定是回复还是私信

阶段产出：

- 新的 `OperationRecord`
- 可能新增 `BilibiliComment`

#### 这套 `Strategy` 生成的 `TaskSpec` 会是什么样

当 `Product + Strategy` 确定之后，就可以生成一份具体任务文档，也就是 `TaskSpec`。

这份 `TaskSpec` 里通常会写清楚：

- 本次任务推广的是哪个产品
- 本次任务使用的是哪套策略
- 本次任务由哪些账号执行
- 阶段 0 到阶段 4 分别要做什么
- 每个阶段调用哪些 `Capability`
- 当前哪些步骤已完成，哪些还未完成
- 如果中断了，下次从哪里继续

#### 这个例子里三者分别在做什么

- `Capability`
  - 提供“找视频”“找意向用户”“持续监听”这些能力
- `Strategy`
  - 定义这套推广为什么这样推进，为什么先监听、为什么先找视频、为什么先公开触达、为什么再继续跟进
- `TaskSpec`
  - 把这次“AI 编程训练营推广任务”真正写成一份可执行文档

---

## 3. `TaskSpec`

`TaskSpec` 是基于 `Product + Strategy` 生成的一份具体任务文档，建议用 md 形式。

它不是抽象模板，而是一份真正要执行的任务说明。

### `TaskSpec` 主要承载什么

- 这次任务的目标
- 这次任务对应哪个 `Product`
- 这次任务使用哪个 `Strategy`
- 这次任务由哪些 `Account` 执行
- 这次任务分几个阶段
- 每个阶段要调用哪些 `Capability`
- 当前的 checklist
- 暂停和恢复说明

### 为什么适合用 md

因为它需要：

- 可读
- 可改
- 可讨论
- 可打钩
- 可暂停
- 可恢复

所以它更适合作为一份任务文档，而不是做成一个很重的数据库对象。

---

## 三者的关系

可以直接理解成：

```text
Product + Strategy -> TaskSpec
TaskSpec -> calls -> Capability
```

再展开一点就是：

```text
Product
  -> 提供业务上下文

Strategy
  -> 提供阶段流程和执行规则

TaskSpec
  -> 把这次任务真正写出来

Agent
  -> 按 TaskSpec 执行
  -> 在执行过程中调用 Capability
```

---

## 和数据层的边界

这里也需要明确一下边界：

- `TaskSpec` 负责说明“这次任务怎么跑”
- 领域实体负责存平台对象
- `OperationRecord` 负责存关键操作记录

也就是说：

- 任务说明在 `TaskSpec`
- 真实操作事实在 `OperationRecord`
- 视频、用户、评论这些事实在领域实体

---

## 定案

二期上层编排统一按下面理解：

- `Capability` 是封装好的能力包
- `Strategy` 是带细节说明的策略模板
- `TaskSpec` 是基于 `Product + Strategy` 生成的具体任务文档
- `TaskSpec` 用 md 形式来承载
- agent 基于 `TaskSpec` 持续执行
- 执行过程中产生的关键操作，统一沉淀到 `OperationRecord`
