# bilibili-api-skill 二期指令包设计

## 说明

这份文档只讨论设计层的问题：

- 什么是 `Command`
- 什么是 `Capability`
- 什么是指令包
- 它们和 `Strategy` 的关系是什么

这里不讨论：

- 具体有哪些指令
- 每条指令怎么命名
- 最终要落多少个功能包

目标是把这一层概念理清楚。

---

## 我们现在的理解

二期这里需要分清楚 3 层：

1. `Command`
2. `Capability`
3. `Strategy`

一句话理解：

- `Command` 是原子化指令
- `Capability` 是基于指令集再包装一层的功能包
- `Strategy` 是更上层的推广策略模板

也就是说：

- 指令更底层
- 功能包更偏高层功能模块
- 策略更偏推广打法和任务流程

---

## 1. `Command`

`Command` 是最小、最稳定、最清晰的调用单元。

它更偏原子化。

一个指令通常只做一件明确的事。

所以可以把 `Command` 理解成：

- 用户可以直接调用的稳定入口
- agent 也可以直接调用的稳定入口

### `Command` 的特点

每一条指令都应该：

- 作用单一
- 输入清楚
- 输出清楚
- 适合重复调用
- 能被 agent 稳定理解和路由

### `Command` 的定位

`Command` 这一层的价值，不是“复杂”，而是“稳定”。

也就是说，更复杂的能力最终都要落到一批稳定的指令上。

---

## 2. `Capability`

`Capability` 是功能包。

它不是简单的“几条指令拼在一起”，而是在指令集之上再包装出来的一层高层功能模块。

这里最关键的一点是：

- `Capability` 不是脱离 agent 自动完成的
- 它需要 agent 参与执行和判断
- agent 会根据上下文去决定如何更好地调用其中的指令

所以更准确地说：

- 指令提供稳定的执行工具
- 功能包提供更复杂的目标功能
- agent 负责把这些指令组织起来，把功能真正做出来

### `Capability` 的定位

更容易理解地说：

- `Command` 更像单个工具
- `Capability` 更像一套工作方法

也就是说，功能包不是为了替代指令，而是为了告诉 agent：

- 在完成某个复杂功能时
- 哪些指令通常要一起用
- 调用这些指令时要关注哪些判断点

### `Capability` 需要说明清楚什么

一个功能包至少需要写清楚：

- 这个功能是做什么的
- 它通常会使用到哪些 `Command`
- agent 在执行时需要重点判断什么
- 它的输入是什么
- 它的输出是什么
- 它适合在什么场景下使用

---

## 3. `Strategy`

`Strategy` 是更上层的推广策略模板。

它不会直接定义单条指令怎么写，而是会说明：

- 这套推广的目标是什么
- 这套推广的整体打法是什么
- 某个阶段应该用哪个 `Capability`
- 某个功能包在什么阶段使用
- 整个任务应该按什么方式推进

这里要特别明确：

`Strategy` 不是简单的功能包集合。

它更像是一套完整的推广心法和流程模板。

所以三层关系可以直接理解成：

```text
Strategy
  -> uses -> Capability

Capability
  -> uses -> Command
```

---

## 4. 为什么需要“指令包”这一层

如果没有清晰的指令层，后面会有两个问题：

### 1. agent 会过度自由发挥

如果只有能力和策略，没有稳定指令入口，agent 很容易在执行时：

- 自己拼流程
- 自己猜调用方式
- 自己脑补下一步

这样最终就会重新变黑盒。

### 2. 用户和 agent 的入口不统一

如果没有清晰指令，用户看 skill 时很难知道：

- 到底有哪些可以直接调用的入口
- 哪些能力是稳定的
- 哪些动作是系统明确支持的

所以这一层的作用其实很明确：

- 对用户来说，它是可调用入口
- 对 agent 来说，它是可路由入口

---

## 5. 指令包应该覆盖什么

这套指令包，应该覆盖两类内容：

### 第一类：B 站能力相关指令

这类指令主要承接 B 站侧的能力。

比如：

- 登录
- 搜索
- 查视频
- 查评论
- 发评论
- 查私信
- 发私信
- 查通知

### 第二类：Skill 运行相关指令

这类指令不是 B 站原生能力，而是 skill 自身为了稳定运行必须要有的入口。

比如：

- 初始化运行环境
- 查看当前产品
- 查看当前策略
- 查看当前任务
- 查看路径和状态

这类指令同样重要，因为 agent 和用户在实际使用 skill 时，会大量依赖它们。

---

## 6. 一个简单例子

下面用一个简单例子说明这三层的关系。

假设现在有一个目标：

- 根据某个产品，搜索一批相关视频

如果从三层来看：

### `Command`

这一层提供的是原子化动作。

比如：

- 读取产品信息
- 搜索视频
- 查看视频详情

### `Capability`

这一层提供的是高层功能。

比如：

- “视频发现”功能包

这个功能包不会自己凭空完成任务，而是 agent 参考这个功能包的说明，再去组织若干条底层指令。

### `Strategy`

这一层决定的是：

- 这个“视频发现”功能，应该在整个任务的哪个阶段被调用

所以更直白地说：

- `Command` 解决“怎么做一个动作”
- `Capability` 解决“怎么完成一个功能”
- `Strategy` 解决“这套推广应该怎么打”

---

## 7. 定案

二期指令包统一按下面理解：

- `Command` 是最小、最稳定的原子化指令
- `Capability` 是基于指令集再包装一层的高层功能包
- `Capability` 需要 agent 参与执行和判断
- `Strategy` 在更上层定义推广打法，并决定功能包如何服务这套打法
- 指令包需要同时覆盖：
  - B 站原生能力相关指令
  - Skill 自身运行相关指令
- 每一条指令都需要说明清楚：
  - 作用
  - 输入
  - 输出
  - 什么时候用
  - 常见下一步
