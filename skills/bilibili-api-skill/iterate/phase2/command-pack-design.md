# bilibili-api-skill 二期指令包设计

## 说明

这份文档只说明三个概念：

- 什么是 `Command`
- 什么是 `Capability`
- 什么是 `Strategy`

---

## 三层关系

二期统一按下面理解：

- `Command`
  - 原子化指令
- `Capability`
  - 基于指令集再包装一层的功能包
- `Strategy`
  - 更上层的推广策略模板

可以直接这样理解：

- `Command` 解决“做一个动作”
- `Capability` 解决“完成一个功能”
- `Strategy` 解决“这套推广怎么打”

---

## 1. `Command`

`Command` 是最小、最稳定的调用单元。

一条指令只做一件明确的事。

它是：

- 用户可以直接调用的入口
- agent 也可以直接调用的入口

它的特点是：

- 单一
- 清楚
- 稳定
- 可重复调用

---

## 2. `Capability`

`Capability` 是功能包。

它不是简单的指令列表，而是在指令集之上再包装出来的一层高层功能。

它的特点是：

- 目标更完整
- 会用到多条指令
- 需要 agent 参与执行和判断

在主副 `Agent` 模式下，`Capability` 还可以被理解成：

- 主 `Agent` 可以直接派给副 `Agent` 的标准工作包
- 副 `Agent` 在内部自行组织多条 `Command` 完成这项工作
- 主 `Agent` 只回收结果，不需要一直阻塞在这个过程里

可以把它理解成：

- 指令是工具
- 功能包是用这些工具去完成一项功能的方法

一个功能包至少要说明清楚：

- 这个功能是做什么的
- 会用到哪些指令
- agent 需要重点判断什么
- 输入是什么
- 输出是什么

---

## 3. `Strategy`

`Strategy` 是更上层的推广策略模板。

它不是简单的功能包集合，而是一整套推广打法。

它需要说明：

- 目标是什么
- 整体打法是什么
- 分几个阶段
- 每个阶段用哪些功能包
- 哪些动作优先
- 哪些动作需要克制
- 在什么情况下推进、暂停或停止

所以三层关系可以直接理解成：

```text
Strategy
  -> uses -> Capability

Capability
  -> uses -> Command
```

---

## 定案

二期指令包统一按下面理解：

- `Command` 是最小、最稳定的原子化指令
- `Capability` 是基于指令集再包装一层的高层功能包
- `Capability` 需要 agent 参与执行和判断
- `Capability` 也适合作为主 `Agent` 派给副 `Agent` 的标准任务单元
- `Strategy` 在更上层定义推广打法，并决定功能包如何服务这套打法
