# 任务书：SQLite 事实层与关键操作记录

## 1. 任务综述

### 1.1 任务定义

围绕 `Account`、`Product`、`BilibiliUser`、`BilibiliVideo`、`BilibiliComment`、`OperationRecord` 建立新的 `SQLite` 事实层。

### 1.2 核心目标

这个任务主要解决两件事：

- 系统事实往哪里落
- 关键动作怎么稳定回溯

### 1.3 当前状态

- 已完成
  - 二期核心领域对象已经在设计文档中收口
  - `SQLite` 方案已经定案为 `Node 22.13+` + `node:sqlite`
  - `OperationRecord` 已经明确为统一关键操作记录
- 未开始
  - 新系统的事实层还没有正式建立
  - 新系统的查询口径和写入口径还没有落地
- 参考边界
  - 旧版日志、旧版 JSON、旧版状态文件不作为二期权威数据源

### 1.4 重要性与紧急程度

- 优先级：`P0`
- 说明：这是二期可观察、可回溯、可判断的基础

### 2.2 这份任务主要产出什么

这份任务主要要把新的事实层规则定清楚：

- 平台对象和系统对象有哪些
- 关键操作如何统一记录
- 什么信息放数据库，什么信息继续放文件
- 查询能力如何支持后续策略执行和任务恢复

这层是二期的新数据底座，不是对旧版日志的补丁式整理。

### 2.3 本任务范围

包括：

- 领域对象的事实层落位
- `OperationRecord` 的记录范围和查询口径
- 历史记录、去重判断、风控辅助查询
- 数据库与产品文件、任务文件的边界

不包括：

- 用数据库承载 `TaskSpec`
- 图形化报表
- 复杂数据分析模型

## 3. 注意事项

### 3.1 已知风险

- 如果事实层没有先立住，主副 `Agent` 仍然会回到“猜历史、猜状态”的老问题
- 如果动作记录继续散落在日志里，验收和回溯都会持续困难

### 3.2 强约束

- 数据库只存事实，不存长篇任务内容
- `OperationRecord` 统一承载评论、评论回复、私信等关键动作
- 产品详细资料和任务详细内容继续走文件

### 3.3 耦合点

- 任务 4 的指令层需要写入这里
- 任务 5 的策略和任务体系需要读取这里
- 任务 6 的执行闭环和恢复逻辑依赖这里

### 3.4 避坑

- 不要继续把日志当权威事实层
- 不要把关键动作拆成多个重复概念
- 不要把任务说明和平台事实混存

## 4. 验收标准与交付物

- [ ] 二期核心领域对象已经有统一事实层承载
- [ ] `OperationRecord` 已经成为关键操作的统一记录入口
- [ ] 可以按账号、动作类型、目标对象做稳定查询
- [ ] 可以为去重、跳过、继续跟进提供直接依据
- [ ] 上层任务和策略可以直接消费这套事实层
