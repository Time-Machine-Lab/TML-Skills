# 任务书：B 站适配层与原子指令层

## 1. 任务综述

### 1.1 任务定义

围绕二期核心接口能力，建立新的 B 站适配层和原子指令层。

### 1.2 核心目标

这个任务主要解决三件事：

- 二期到底需要哪些 B 站原子能力
- 这些能力怎么收成稳定指令
- 指令结果怎么让上层稳定消费

### 1.3 当前状态

- 已完成
  - 二期核心接口能力已经梳理完成
  - 搜索能力的入参口径已经明确参考 `scripts/bilibili-mcp-lite.mjs`
- 未开始
  - 新系统的适配层和指令层还没有正式建立
  - 新系统的统一输入输出口径还没有落地
- 参考边界
  - 旧版脚本只可参考少量 B 站接口行为、请求参数和字段形态
  - 旧版命令体系、旧版流程语义、旧版模块划分不作为二期指令层基础

### 1.4 重要性与紧急程度

- 优先级：`P0`
- 说明：这是二期唯一允许直接参考旧版脚本细节的工作包

### 2.2 这份任务主要产出什么

这份任务主要要把新的底层能力体系定清楚：

- B 站适配层负责直接对接平台接口
- 原子指令层负责把底层动作变成稳定入口
- 上层功能包和策略只消费新的指令层，不直接绑定底层请求细节

这里的“参考旧版”只针对少量接口行为，不代表沿用旧版架构。

### 2.3 本任务范围

包括：

- 登录与会话相关能力
- 账号信息相关能力
- 视频搜索与视频详情相关能力
- 评论、通知、私信相关能力
- 原子指令的输入输出口径
- 统一错误语义、风控提示和下一步指引

不包括：

- 功能包逻辑
- 策略编排逻辑
- 主副 `Agent` 调度逻辑

## 3. 注意事项

### 3.1 已知风险

- 如果把旧版命令继续搬过来，二期虽然换了名字，本质上还是旧系统
- 如果指令只返回原始日志，上层 `Agent` 还是需要自己猜下一步

### 3.2 强约束

- 一条 `Command` 只做一个明确动作
- 适配层和指令层按新系统重新设计，不承接旧模块边界
- 指令输出要简洁、稳定、可被 `Agent` 直接消费
- 关键结果要能回写到新的事实层

### 3.3 耦合点

- 依赖任务 2 提供新的运行空间
- 需要和任务 3 对齐数据写入口径
- 任务 5 和任务 6 都会直接消费这里的能力

### 3.4 避坑

- 不要在指令层夹带策略流程
- 不要为了复用旧代码而保留旧命令语义
- 不要让指令结果缺少风控和下一步信息

## 4. 验收标准与交付物

- [ ] 二期核心 B 站原子能力已经收成新的适配层
- [ ] 二期原子指令的输入输出口径已经统一
- [ ] 指令结果里已经包含清晰的下一步指引和限制提示
- [ ] 关键结果可以稳定对接新的事实层
- [ ] 上层功能包、策略包和主副 `Agent` 可以直接消费这套能力
