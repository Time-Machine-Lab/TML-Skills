# 策略：基础评论-回复-私信闭环

## 定位

这是一套面向 B 站场景的基础增长运营策略。

它更像是一套增长心法模板，而不是几个功能包的简单拼接。

它会统一规定：

- 当前任务应该先做什么、后做什么
- 每个阶段调用哪个功能包
- 每个阶段允许使用哪些指令与动作边界
- 每个阶段的判断重点是什么
- 每个阶段对外表达应遵循什么提示词导向
- 公开触达、回复承接、私信升级分别怎么把控

主路径：

`视频发现 -> 评论区线索挖掘 -> 公开评论触达 -> 评论回复承接 -> 高意向私信升级`

## 何时使用

- 产品已经具备基础资料，希望在 B 站跑通第一条增长闭环时使用
- 希望以较稳妥、可审核、可回溯的方式推进公开触达和后续承接时使用
- 当前阶段更看重完整闭环、风险控制和可复盘能力，而不是纯规模冲量

## 前置条件

- 已完成运行环境初始化和账号登录
- 产品资料已经能说明目标用户、核心卖点和禁止触碰的边界
- 当前任务允许以阶段推进的方式执行，并接受“先审核后执行（`review-first`）”
- 任务执行者接受“先公开评论，再回复承接，再高意向私信”的主路线

## 核心构成

- 功能包：
  - `video-discovery`
  - `comment-prospect-mining`
  - `public-comment-outreach`
  - `reply-and-dm-follow-up`
- 指令：
  - 产品查询类指令
  - 视频搜索和详情查询类指令
  - 评论扫描与评论发送类指令
  - 通知巡检与私信收发类指令
  - 关键动作记录 `OperationRecord` 的去重与记录指令
- 提示词导向：
  - 评论要自然进入讨论
  - 回复要基于上下文继续承接
  - 私信只用于高意向升级

## 全局工作规则

- 先建立目标视频池，再进入评论区线索识别，不要一上来就外发。
- 公开区优先建立自然互动，不做强导流。
- 评论回复承接优先于私信升级。
- `reply-and-dm-follow-up` 不是默认常开能力，只有在公开触达后出现互动信号，或任务明确进入跟进窗口时才打开。
- 所有真实外发动作都必须写入关键动作记录 `OperationRecord`。
- 所有真实外发动作在发送前，都必须先调用 `records.cooldown-check`，以返回的近期记录和当前生效规则为准。
- 每一条真实外发动作都必须补一段简短说明，说明发送依据、目的或意义；最好控制在 100 字内。
- 具体节流数值不写在策略模板里；如需调整，统一用 `records cooldown-policy-set` 更新中心规则。
- 主 Agent 负责任务主控、阶段判断和状态回写。
- 副 Agent 只在被派发的能力包或阶段片段内执行。

## 阶段打法

### 阶段 1：`discover_videos`

- 功能包：`video-discovery`
- 目标：找到一批与产品和用户场景相关的视频来源
- 提示词重点：围绕产品卖点、用户问题和使用场景去提炼关键词，不要只追求大词流量
- 产出：候选视频列表与筛选说明

### 阶段 2：`mine_prospects`

- 功能包：`comment-prospect-mining`
- 目标：从目标视频评论区识别更有意向的用户与线索
- 提示词重点：优先找求助、求推荐、比较、真实问题表达，不要把所有评论都当线索
- 产出：线索列表与判断依据

### 阶段 3：`public_comment_outreach`

- 功能包：`public-comment-outreach`
- 目标：在适合的视频下发出第一轮公开评论触达
- 提示词重点：评论要自然、短、贴上下文，像参与讨论而不是发布广告
- 产出：评论发送结果、跳过说明、操作记录

### 阶段 4：`reply_follow_up`

- 功能包：`reply-and-dm-follow-up`
- 目标：在合适时机打开跟进能力，承接公开评论触达后产生的评论回复与互动
- 提示词重点：顺着用户当前问题继续聊，优先在评论区承接，不要过早转私信
- 产出：跟进结果、高意向候选、操作记录

### 阶段 5：`high_intent_dm`

- 功能包：`reply-and-dm-follow-up`
- 目标：在高意向时机打开私信升级，只对高意向用户做私信升级
- 提示词重点：只有用户已经表现出明确需求、主动追问或更强信号时，才允许转入私信
- 产出：私信升级结果、跳过原因、操作记录

## 提示词导向

- 对外表达要像真人讨论，不要像模板化营销文案。
- 优先使用产品相关、场景相关、问题相关的话术，而不是一上来就堆卖点。
- 评论和回复都应短一些，给用户留继续互动的空间。
- 私信不是默认动作，而是高意向阶段的受控动作。

## 审核规则

- 所有外发阶段默认走“先审核后执行（`review-first`）”。
- 新的外发阶段首轮必须先审核。
- 审核通过后，同阶段后续动作可以继续推进，但仍要受风控、去重和节流限制。

## 主 Agent 职责

- 选择当前阶段是否继续、本地执行还是派发给副 Agent
- 决定什么时候进入审核、什么时候进入下一阶段
- 维护 `TASK.md`、`WORKLOG.md` 和阶段主状态
- 汇总结果、异常和恢复路径

## 副 Agent 职责

- 在被派发的能力包或阶段片段范围内完成工作
- 严格遵守当前阶段目标与边界
- 如果执行真实外发动作，立即写入关键动作记录 `OperationRecord`
- 如果执行真实外发动作，必须同时写下简短说明
- 如果执行真实外发动作，执行前必须先完成节流检查
- 把结果交还主 Agent 做统一回写

## 建议做法

- 先找视频，再找评论区线索
- 公开区先建立自然互动
- 高意向才升级私信
- 在每个阶段结束后形成可复盘的产物和说明

## 禁止事项

- 不要直接把私信当第一触点
- 不要绕过关键动作记录 `OperationRecord`
- 不要绕过外发前的节流检查
- 不要让副 Agent 直接改任务主控状态
- 不要把“广撒网”和“精准承接”混成一个模糊动作
- 不要在没有审核或没有高意向依据的情况下强行推进下一层动作

## 完成标志

- 任务已经完成视频来源准备、线索识别、公开触达、回复承接和高意向私信升级的完整闭环
- 每个阶段都有明确输出、跳过说明或失败说明
- 所有真实动作都能回查到相应关键动作记录 `OperationRecord`

## 受阻处理

- 如果产品资料不足以支撑阶段推进，先补产品资料，不要硬跑策略
- 如果某一阶段结果质量明显不足，应在当前阶段修正，不要带着低质量结果进入下游
- 如果遇到风控、权限、审核未通过或上下文不足，立即停在当前阶段并交回主 Agent
