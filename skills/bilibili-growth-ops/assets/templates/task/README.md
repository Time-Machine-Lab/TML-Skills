# 任务模板说明

这个目录不是“业务任务模板库”，而是任务运行文档模板库。

它负责统一任务目录里几类关键文档的格式，让主 Agent 和副 Agent 可以围绕同一套文档骨架协作。

## 统一边界

- `Strategy` 决定这套推广打法怎么打
- `Task` 决定这一次具体怎么做
- 这个目录只负责把任务运行过程落成统一格式的文档

也就是说，这里不会写死某个产品的具体任务内容。

真正的任务步骤、批次安排、补充说明、异常处理、阶段内工作文件，都应该由 Agent 基于“产品资料 + 策略模板”在任务目录里继续生成。

## 4 个模板分别做什么

### `TASK.md.tmpl`

任务主控卡。

作用：

- 展示当前任务的总目标
- 展示阶段清单和当前焦点
- 作为主 Agent 观察任务状态的总入口

它更像“任务控制面板”，不是完整执行细则。

### `WORKLOG.md.tmpl`

任务时间线日志。

作用：

- 用最短的话记录关键变化
- 记录任务创建、派工、审核、回写、暂停、恢复等事件
- 方便后续快速回看任务过程

它适合记录“发生了什么”，不适合承载大段计划说明。

### `DELEGATION.md.tmpl`

副 Agent 派工单。

作用：

- 明确本次派工的范围和边界
- 说明当前阶段的策略要求
- 说明本次允许调用哪些指令
- 说明做完之后应该如何回交结果

它更像“标准任务单”。

### `REVIEW.md.tmpl`

外发阶段审核单。

作用：

- 在首轮外发前做审核
- 明确当前阶段为什么需要审核
- 让主 Agent 或人工快速判断是否允许继续

它更像“审核卡片”。

## 是否能高效率完成任务

可以，但要注意它的定位。

这套模板擅长的是：

- 统一任务文档结构
- 降低主副 Agent 协作成本
- 让任务可暂停、可恢复、可审核、可回溯

它不擅长的是：

- 直接替代 Agent 生成具体任务内容
- 直接写死所有产品场景下的执行细节

所以更准确地说：

- 它很适合做“高效率的任务运行壳”
- 但不应该被当成“完整任务内容本身”

## 推荐使用方式

建议这样理解：

1. `task create` 先生成任务容器
2. 主 Agent 基于“产品资料 + 策略模板”补充这次任务的具体执行内容
3. 需要派工时生成 `DELEGATION.md`
4. 需要审核时生成 `REVIEW.md`
5. 过程中持续更新 `TASK.md` 和 `WORKLOG.md`

这样效率会比较高，而且不会把任务写死。
