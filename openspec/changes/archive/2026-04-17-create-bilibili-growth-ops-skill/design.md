## Context

`bilibili-growth-ops` 是一个全新的 skill，不是 `bilibili-api-skill` 的继续重构。二期的目标是把零散接口能力收成一套完整的增长运营系统，核心包括：

- 新的主 Skill 入口和跨 agent 可用的运行空间
- 产品库与产品知识提炼
- SQLite 事实层与 `OperationRecord`
- B 站适配层与原子指令层
- `Capability` / `Strategy` / `TaskSpec`
- 主副 `Agent` 协作闭环

当前已经有完整的 phase-2 设计文档，但这些内容还没有被收成一个新的 skill 包，也没有被正式写成实现前的 OpenSpec 契约。

这个 change 需要同时处理 3 类约束：

1. 不能再把旧 skill 当成工程基础  
2. 运行路径不能绑定 OpenClaw，必须对不同 agent 环境都成立  
3. 首版要追求完整闭环，但仍然需要把风险控制在单账号、单策略、阶段级审核之内

## Goals / Non-Goals

**Goals:**

- 定义一个新的 skill 包 `skills/bilibili-growth-ops/`
- 建立新的主 Skill、运行空间、产品库、事实层、指令层和任务编排体系
- 让任意产品资料都能进入产品库，并被 `Agent` 提炼出可推广点
- 让首版能够跑通“评论触达 -> 回复承接 -> 高意向私信升级”的完整闭环
- 建立主副 `Agent` 协作方式，但保持回写边界清楚、恢复路径清楚
- 在正式实现前，把新的命令面和 SQLite 存储契约补进 `docs/api/` 和 `docs/sql/`

**Non-Goals:**

- 不兼容旧版 `bilibili-api-skill` 的运行目录、命令体系和任务链路
- 不处理多账号并发执行
- 不做复杂图形化页面
- 不在首版就接系统钥匙串
- 不把产品模板一次性写死成最终版本

## Decisions

### 1. 新 skill 独立建包，不在旧 skill 上继续叠加

决定：

- 新 skill 目录名定为 `skills/bilibili-growth-ops/`
- 中文名定为 `B站增长运营系统`
- 展示名定为 `Bilibili Growth Ops`

原因：

- 二期的系统边界已经远超过“API skill”
- 如果继续沿用旧包名和旧包结构，后续实现很容易继续受旧心智和旧目录拖拽

备选方案：

- 继续在 `bilibili-api-skill` 上演化  
  不选，因为会持续混淆“旧工具”和“新系统”的边界

### 2. 运行空间使用通用 `runtime-root`，不绑定任何特定 agent 路径

决定：

- 默认运行目录使用 `~/.tml/skills/bilibili-growth-ops`
- 同时允许通过参数或环境变量覆盖 `runtime-root`
- 所有运行态数据、任务文件、产品库和 SQLite 文件都统一放在这里

原因：

- 这是最通用的跨 agent 方案
- 不依赖 OpenClaw、Codex 或其他具体宿主路径
- 后续更容易迁移和备份

备选方案：

- 绑定 OpenClaw 或旧 skill 运行目录  
  不选，因为与“任意 agent 都能安装使用”的目标冲突

### 3. 产品库采用“文件主导 + 可提炼”的结构

决定：

- 产品详细资料使用文件承载
- 首版产品模板只定义初版结构，不锁死最终字段
- 产品库不仅保存原始产品信息，也允许保存初步提炼结果和素材索引

原因：

- 产品模板后续一定会迭代
- 如果首版就把产品字段彻底写死，后续会频繁破坏兼容性
- 文件结构更适合人工补充、协作编辑和素材管理

备选方案：

- 产品信息重度结构化入库  
  不选，因为当前阶段产品资料高度开放，过早固化会增加维护成本

### 4. 事实层和任务层分离

决定：

- SQLite 只存结构化事实
- `Account`、`Product`、`BilibiliUser`、`BilibiliVideo`、`BilibiliComment`、`OperationRecord` 进入事实层
- `TaskSpec`、`TASK.md`、`WORKLOG.md`、阶段输出继续走文件

原因：

- 事实层和任务层职责不同
- 这样既能稳定回溯，又能保留任务文档的可读、可改、可暂停能力

备选方案：

- 把任务状态也重度入库  
  不选，因为任务结构仍然高度依赖 `Agent` 判断和文档协作

### 5. B 站适配层只复用低层请求知识，不复用旧架构

决定：

- 旧 skill 只允许提供少量 B 站请求行为参考
- 新 skill 重新建立适配层和原子指令层
- 指令输出必须包含结构化结果、风险提示和下一步指引
- 评论、回复、私信等真实外发动作的节流规则只允许存在一个中心配置源，并支持被 `Agent` 精确查询和更新

原因：

- 旧 skill 的问题主要不在“有没有请求能力”，而在“整个系统组织方式不成立”
- 新指令层必须面向 `Capability`、`Strategy` 和主副 `Agent` 协作来设计
- 如果节流规则分散在策略、功能包和动作指令里，`Agent` 很容易产生幻觉，也不利于后续维护与风控

备选方案：

- 直接搬旧命令体系  
  不选，因为会把旧流程语义一并带进来

### 6. 首版只做一套内置基础策略，先把闭环跑通

决定：

- 首版默认只做一套基础策略
- 路径固定为：评论触达 -> 评论回复承接 -> 高意向私信升级
- 执行模式默认 `review-first`
- 审核粒度定为“阶段级审核 + 首轮必审”

原因：

- 一套打透，比多套都浅更适合首版
- 风险更可控，也更容易验收

备选方案：

- 首版就同时提供多套策略  
  不选，因为会分散实现和验证注意力

### 7. 主副 `Agent` 采用“任务片段派发”，而不是“单动作派发”

决定：

- 主 `Agent` 负责规划、判断、派发和总览回写
- 副 `Agent` 承接一个 `Capability` 或一个阶段片段
- 副 `Agent` 不被定义成一步命令执行器

原因：

- 如果副 `Agent` 只执行单次动作，主 `Agent` 仍然会被细节阻塞
- 任务片段派发更符合功能包和阶段执行的结构

备选方案：

- 副 `Agent` 只负责单次动作  
  不选，因为收益太低，仍然无法明显改善长任务阻塞

### 8. 写入边界采用“动作即记事实，主控统一收状态”

决定：

- 副 `Agent` 如果执行了真实外部动作，要立即写对应 `OperationRecord`
- 主 `Agent` 统一收口 `TASK.md`、`WORKLOG.md` 的主状态
- 副 `Agent` 不负责修改策略、产品主资料和任务主控状态

原因：

- 可以同时保证动作记录不丢失、任务主状态不失控
- 更适合审计、恢复和阶段级审核

备选方案：

- 所有写入都交给主 `Agent`  
  不选，因为会让主 `Agent` 成为所有回写瓶颈

- 副 `Agent` 自行维护任务主状态  
  不选，因为容易出现多处状态源不一致

### 9. 首版先使用本地受限文件存储会话与凭证

决定：

- 会话文件放在 `runtime-root`
- 文件权限尽量收紧
- 不在首版就接系统钥匙串

原因：

- 这是最通用的跨环境方案
- 不会额外绑定宿主能力

备选方案：

- 首版就接系统钥匙串  
  不选，因为会增加环境耦合和实现复杂度

## Risks / Trade-offs

- [产品模板仍在演进] → 首版只定初版结构，并把“可扩展字段”和提炼结果放在文件层处理
- [阶段级审核可能降低自动化速度] → 先优先保证风险可控，后续再逐步提升自动化等级
- [单账号会限制吞吐] → 首版接受吞吐受限，先把闭环、回溯和风控做稳
- [旧 skill 与新 skill 会在仓库中并存一段时间] → 在设计和实现中明确“新 skill 独立建包，旧 skill 不再作为基础”
- [docs/api 与 docs/sql 当前为空] → 在 implementation tasks 中把命令契约和 SQLite 契约列为最先完成的前置任务

## Migration Plan

1. 在 `openspec` 中完成 proposal、design、specs、tasks，明确新 skill 的边界和契约
2. 先补 `docs/api/` 和 `docs/sql/` 中与新 skill 对应的命令面、存储模型契约
3. 在 `skills/` 下创建新的 `bilibili-growth-ops` 包和基础目录
4. 逐步实现运行空间、产品库、事实层、指令层、策略任务体系和主副 `Agent` 协作
5. 保持旧 skill 不动，不做运行态迁移依赖

回滚策略：

- 因为这是新包建设，首版回滚只需要停止使用 `bilibili-growth-ops`，旧 skill 不受影响

## Open Questions

- 产品模板初版的具体栏目顺序仍然可以继续优化，但这不阻塞新 skill 的骨架设计
- 后续是否要支持网页抓取、文档解析、富媒体自动提炼，可以作为后续 change 再扩展
